# Desktop app — Electron shell with game-style launch

**Branch:** feature/desktop · **Date:** 2026-07-21 · **Author:** Denis (+ Claude)

## What changed
First slice of NilBot as a desktop app (`electron/`): `npm run desktop`
builds the adapter-node bundle and launches Electron. Launch feels like
a game, not a webapp: a frameless splash (glowing d20 in a slow rune
ring, "NILBOT — the DM's workbench", "Setting the table…") while the
server wakes, then a menu-bar-less 1440×900 window. Campaign data lives
in the OS user-data folder (`~/.config/NilBot` on Linux,
`%APPDATA%/NilBot` on Windows) — the server is forked with that cwd, so
zero storage-code changes. The server binds 0.0.0.0 so players on the
LAN can still open Battle Ready/canvas in a browser. External links
open in the real browser; single-instance lock; closing the window
quits.

Dev runs the server on system node (repo natives stay built for plain
node); packaged builds will run it through Electron's utilityProcess
with electron-builder rebuilding natives at package time —
`electron-builder.yml` is staged (AppImage/NSIS/dmg, Game category) but
the builder isn't wired into CI yet. When Steam gets real, a separate
NilBot-Tabletop repo will hold store assets/packaging CI and consume
tagged releases of this repo; this repo stays the single source of app
code.

## Why
NilBot should eventually sit in a Steam library — a DM's toolbox
presented like a game. Also: a real desktop app is nicer at the table
than a terminal + browser tab.

## How to test
`npm run desktop` on a machine with a display: splash appears, then the
app window with first-run DM setup (fresh data dir in userData — your
repo data/ is untouched). Verified headless in CI-style: the built
server boots from an arbitrary cwd, creates its own data dir, serves
first-run setup.

## Deploy steps
None — the web/live pipeline is untouched.
